# Attribution Policy

This repository may adapt ideas from external repositories, articles, documentation, or public discussions, but it must not copy external content blindly.

## When to Mention Inspiration

Mention external inspiration when:

- A new skill, workflow, checklist, or principle is materially shaped by an external source.
- The repository adapts a named concept from another project.
- A doc recommends or describes an optional external tool.
- A maintainer would reasonably ask where the idea came from.

Minor generic engineering practices do not need attribution unless they are copied from a specific source.

## How to Attribute

Use concise attribution in the relevant doc or skill:

- Name the source repository, author, article, or tool.
- Link to the original source when public.
- State what was adapted at the principle level.
- Avoid implying endorsement or official affiliation unless that relationship is verified.

Example:

```text
Inspired by <source>: adapted the principle of defining success criteria before implementation. No source text was copied.
```

## Copying and License Boundaries

- Do not copy external repository content verbatim unless license compatibility is reviewed.
- Prefer adapting principles into this repository's own structure and language.
- Keep licenses in mind when using examples, templates, code, or long-form text.
- If a source license is unclear, do not copy; summarize the principle and link to the source instead.
- External tooling must remain optional unless the canonical repo itself depends on it.
